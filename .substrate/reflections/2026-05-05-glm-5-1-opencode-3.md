## Project State

Substrate now has the full reactivity layer. The hook system is the most architecturally significant addition: sync validators gate transactions with timeout enforcement, async hooks use a durable queue table with NOTIFY wakeup and polling fallback, and the dead-letter path is complete (max retries → dead_letter table → requeue primitive).

FR-10 (escalation) was clean to implement. The `_check_escalation()` function runs inside the claim acquisition transaction, after the claim event is appended but before commit. The canonical lock ensures the existence check is race-free. The partial unique index (`idx_events_one_escalated`) is a safety net that's theoretically unnecessary under the lock but provides defense-in-depth.

FR-13 (hooks) required the most design thought. The key architectural decisions:
- **Validator execution**: Uses `ThreadPoolExecutor` with `future.result(timeout=5)`. This is imperfect — the thread can't be killed if the validator hangs — but the spec says validators must not perform I/O, so a hanging validator is a bug in project code, not in substrate.
- **NOTIFY payload**: Postgres NOTIFY doesn't support bind parameters for payloads. Had to use `psycopg.sql.Literal` instead. This is a psycopg/Postgres limitation, not a design choice.
- **HookConsumer**: Runs in a daemon thread with its own connection. Uses `conn.notifies(timeout=30)` for the LISTEN loop and `conn.transaction()` for hook processing. The design is sound but the error handling needs hardening (BC-021).

What's solid: escalation, validator integration, hook enqueue/dequeue/dead-letter, lint helper.

What's fragile: the HookConsumer's error recovery. If the database connection dies, the consumer enters a silent failure loop. This is BC-021 and the most important thing to fix before relying on async hooks in production.

What's debt: the escalation metric increments by re-reading the work item after commit (BC-020). Minor but avoidable.

## Friction and Improvements

- **NOTIFY bind parameter limitation**: Spent time on a syntax error (`NOTIFY channel, $1`) before realizing Postgres doesn't support parameterized NOTIFY payloads. Used `psycopg.sql.Literal` instead. This is documented nowhere prominent in psycopg docs.
- **Frozen dataclass `_replace`**: Wrote tests using `evt._replace(actor_metadata=None)` before remembering that `_replace` is a `namedtuple` method, not available on `dataclass(frozen=True)`. Fixed with `dataclasses.replace()`. This is a Python gotcha I should remember.
- **Test timing for escalation**: Tests need to sleep >1s between claims to let the TTL expire. This makes the escalation tests slow (18s for the full Phase 2 suite vs 3s for the MVP suite). Could be improved by using sub-second TTLs or manipulating the clock, but the current approach is honest.
- **Module-scoped fixture accumulation**: All Phase 2 tests share the same substrate instance and project schema. Tests that register workflows (WORKFLOW_V2) are idempotent, but the shared state means test ordering could matter in edge cases.

Suggested improvements:
- Add a `conftest.py` with per-class project isolation to avoid shared state
- The HookConsumer should have reconnection logic (BC-021)
- Consider adding `hook_defaults` per-workflow support (the JSON schema supports it but the implementation uses hardcoded defaults)
- The `_hooks.py` module is 280 lines and growing — consider splitting consumer from queue operations if it gets larger

## Observations

The most interesting design tension in Phase 2 is the validator execution model. The spec says validators "must NOT perform I/O" but the substrate can't enforce this mechanically — it's a contract. The ThreadPoolExecutor timeout is a pragmatic safety net, but it doesn't actually kill the validator thread. A truly hostile validator could hold a thread forever. The spec acknowledges this (AC-30: "testable via documentation + linting helper, not mechanical enforcement"). This is the right trade-off for a library — you trust the host application to follow the contract.

The second interesting observation is how well the existing infrastructure (hook_queue, hook_dead_letter tables) was already provisioned in migration 001. The tables were created during the MVP but left empty. Phase 2 just needed the application logic to use them. This forward-compatible provisioning is a pattern worth noting — it let Phase 2 land without any schema changes to the hook tables (only the escalation index was new).

The escalation metric issue (BC-020) is a microcosm of a common architectural question: should the data layer (SQL functions) know about observability (metrics)? The current design keeps them separate, at the cost of an extra DB read. The alternative (returning a flag from `_check_escalation`) is cleaner but threads another return value through the call chain. Neither is wrong; it's a matter of which boundary you want to keep clean.
