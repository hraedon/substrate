# Session Reflection — 2026-05-16

**Work summary:** Closed out Plan 003 quality gaps: implemented catch-up policy semantics (fire_once/fire_all/skip), fixed InMemory recurrence rule dedup and next_fire_at computation, added fractional second support to ISO-8601 duration parser, wired missing metrics, extracted recurrence API facade from `__init__.py`, and wrote 24 CLI integration tests exercising the admin CLI against real Postgres via subprocess.

## On the project

Substrate feels genuinely complete for a coordination library. The Plan 003 recurrence feature was shipped by the previous agent with correct skeleton code but several untested edge cases (catch-up policies were declared but never consulted, InMemory dedup was overly restrictive). The codebase pattern — pure functions in domain modules, thin transaction-wrapping in `__init__.py` — made the fixes mechanical. The `_contract.py` single-source-of-truth pattern means I didn't have to worry about backend divergence on the new logic; both backends call the same `compute_next_fire` and `_find_next_future_slot`.

The biggest remaining structural concern is `__init__.py` at ~1545 lines. The recurrence extraction shaved ~60 lines, but `transition()` alone is ~180 lines and is the real candidate for extraction if the file continues to grow.

## On the work done

**Catch-up policies** went smoothly. `_find_next_future_slot` is a simple iterative loop with a 10000-iteration guard. The `fire_once` path (skip to next future slot) and `skip` path (advance without firing) are both clean. The `fire_all` path is the simplest — just advance by one slot, consumer loops to catch up. I'm confident this is correct.

**InMemory UUIDv5 fix** was the trickiest part. The InMemory `fire_recurrence` calls `self.create_work_item()` which validates `event_id` as UUIDv4, but recurrence uses `uuid.uuid5()` for deterministic idempotency. Extracted `_create_work_item` with a `skip_event_id_version_check` flag to match how the Postgres backend bypasses validation by calling `_work_items.create_work_item` directly. This is a legitimate internal path — the public API still enforces UUIDv4.

**CLI integration tests** required fighting structlog. The library logs to stdout via `PrintLoggerFactory`, which mixes with `--json` output. The `_extract_json` helper strips structlog lines (timestamp-prefixed) before JSON parsing. This is fragile if structlog format changes, but acceptable for integration tests. The subprocess approach via `python -m substrate._cli` avoids needing the entry point installed.

**`__init__.py` extraction** to `_recurrence_api.py` was straightforward — six thin facade functions that take `(mgr, metrics, project, ...)` instead of `self`. Clean delegation.

## Gaps to flag

- **structlog to stdout in CLI** — `_cli.py` mixes library log output with command output on stdout. For a production CLI, structlog should be configured to write to stderr or be suppressed. `_cli.py:1-376` globally.
- **`_find_next_future_slot` iteration limit** — 10000 iterations at 5-minute intervals covers ~34 days. A rule with 1-second intervals would only cover ~2.7 hours of catch-up. This is documented but could silently lose slots for sub-minute schedules. `_recurrence.py:_find_next_future_slot`.
- **CLI `--json` flag placement** — must come before the subcommand (`substrate --json work-item show`) because it's a top-level argparse argument, not per-subcommand. This is ergonomic but could confuse users expecting `substrate work-item show --json`. `_cli.py:52-56`.
- **No recurrence CLI subcommands** — `register/list/due/fire/cancel/update_recurrence_rule` have no CLI exposure. The API is only accessible programmatically. If operators need to inspect or fire recurrences, they need a Python shell or the planned HTTP sidecar.
