# Session 5 Reflection — 2026-05-05

## What happened

Closed out all 6 remaining open breadcrumbs in one session: BC-022 (content-based workflow idempotency — the main ask), BC-023/024/025 (link payloads, docs, benchmarks — Opus's new filings), and BC-009/016/017 (long-held-open items that needed final verification tests).

## Observations

**Content-based idempotency was the most interesting one.** The original BC-022 had two options: match the spec (reject on re-registration) or amend the spec (allow idempotent). The user chose a third, sharper option: content-based — same content idempotent, different content rejected. This follows the BC-007 precedent (compare payloads on collision) and is the right shape for agent pipelines that retry registration. The spec amendment in §8 was clean.

**Lazy backfill over SQL backfill.** The migration just adds a nullable `content_hash BYTEA` column. Legacy rows get hashed lazily on first access via `register_workflow()`. This avoids the problem of computing JCS hashes in SQL (impossible without PL/Python) or requiring a separate Python backfill step. Simple and self-healing.

**Links are event-sourced, not table-backed.** BC-023's suggested fix said "add payload JSONB column to links table" — but there IS no links table. Links are tracked via `link_created`/`link_removed` events. The payload goes into the event JSONB naturally. No migration needed, which was a pleasant discovery.

**rfc8785 is solid.** The 16 JCS edge-case tests confirmed the library handles float boundaries, integer domain limits, UTF-16 code unit ordering, and negative zero correctly. The only caveat: NFC vs NFD produces different canonical bytes. This is documented but not enforced. In practice, Postgres JSONB stores in NFC, so substrate payloads are safe.

**Scale baselines are healthy.** ~0.46ms/event for replay, ~3ms for link queries, ~914 hooks/sec drain. These are informational baselines for a test container — real hardware will be faster. The benchmarks are excluded from default runs via `@pytest.mark.slow`.

**Breadcrumb count: 0 open, 25 resolved.** Every known issue in the project is closed. The codebase feels clean and ready for v2 consumers.

## Things I'd do differently

The migration backfill was the only design tension point. I initially wrote it as `UPDATE ... SET content_hash = digest(definition::text, 'sha256')` but realized the hash wouldn't match what Python computes via JCS. The lazy-backfill approach is simpler but means the column has NULLs until each row is touched. If someone queries the column directly, they'll see NULLs for untouched rows. This is fine for the current use case — the only consumer is `register_workflow()` — but worth noting.
