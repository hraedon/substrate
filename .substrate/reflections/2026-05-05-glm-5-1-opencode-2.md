## Project State

Substrate is materially more trustworthy after this session. The replay is no longer decorative — it verifies signatures, checks key revocation status, compares all derived fields, and populates the replay table with genuinely replay-derived state. The claims subsystem now produces an audit trail. The API surface has guardrails: you can't bypass FR-11/FR-12 via append_event, and idempotency collisions with different payloads are caught.

What's solid: the event-sourced core (append, sign, project), the replay machinery, the claims lifecycle, the query engine.

What's fragile: the signing scheme (BC-008) promises jsonb-drift survival that the current implementation doesn't deliver — re-verification rebuilds the envelope from deserialized jsonb, which is the exact failure mode the spec says should be defended against. This is a spec-level design decision, not a code bug.

What's debt: BC-007 (unwired idempotency keys) — the parameter is accepted but ignored on most mutations. Either wire it or remove it. The half-state is worse than either choice.

## Friction and Improvements

- **BC-008 blocks BC-009 and partially BC-017**: The signing scheme design decision cascades. You can't fix JCS edge cases (BC-009) or write the AC-26 test (jsonb-drift survival) until the spec chooses between storing canonical envelope bytes vs. HMAC-over-hash. Three breadcrumbs are effectively blocked on one design call.
- **Test isolation**: The session-scoped fixture means all 20 tests share one project. My changes to claims (emitting events) and replay (verifying signatures) could have broken earlier tests. They didn't, but the lack of test isolation is a growing risk. Each breadcrumb fix adds behavior that earlier tests don't anticipate.
- **Breadcrumb README maintenance**: Moving breadcrumbs and updating the README table is error-prone manual work. A script would help, or the README could be auto-generated from frontmatter.

Suggested improvements:
- Add per-test or per-class project isolation (fresh schema per test class)
- Write a `scripts/move-breadcrumb.sh` that handles file move, frontmatter update, and README table rewrite
- The `_claims.py` functions now need `key_set` passed through — this changed the internal API. Consider a context object that carries conn + key_set + project to reduce parameter threading.

## Observations

The breadcrumbs themselves were exceptionally well-written. Each one had: the problem, the spec reference, the exact file and line range, a suggested fix with code sketches, and related items. This is the level of defect reporting that makes resolution fast. I resolved 12 items in one session largely because I didn't have to investigate — the breadcrumbs did that work for the previous reviewer.

The most interesting design tension is between BC-008's two options (store canonical bytes vs. sign the hash). Option A (store bytes) is honest and simple. Option B (sign the hash) is a less standard cryptographic construction. In practice, for substrate's scale, Option A's ~100B per event overhead is negligible. But the spec needs to choose, because the current code does neither effectively — it re-derives the envelope on verification, which is the worst of both worlds.

I was surprised that the claims audit trail (BC-005) was the most invasive change — it touched `_claims.py` (complete rewrite), `__init__.py` (parameter threading), and `_replay.py` (new transition names). The replay cluster was more self-contained. For future work, changes that add event emission to previously-silent operations should be treated as cross-cutting.
