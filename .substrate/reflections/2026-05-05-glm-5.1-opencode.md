# Session Reflection — 2026-05-05

**Model:** glm-5.1
**Invocation:** opencode
**Date:** 2026-05-05

**Work done:** Built the complete substrate MVP from a 708-line Level 3 spec. 3,291 lines of Python across 17 modules, schema-per-project Postgres isolation, RFC 8785 signing, event-sourced state with transactional projections, full claim lifecycle, workflow validation, replay with drift detection, and 20 passing smoke tests.

---

## Project State

The MVP is solid and testable. All core FRs are implemented end-to-end: create, transition, claim, link, query, replay. The code is clean (0 lint errors) and follows the spec closely. The main risk area is the projection update logic in `_events.py` — it's the most complex piece and handles both generic events and transitions. The replay module provides a safety net: any projection bug will surface as drift.

What's fragile: the `_work_items.py` `_rebuild_wf()` function reconstructs a WorkflowDefinition from stored JSONB. If the serialization format drifts (e.g., a migration changes how workflows are stored), this breaks silently. It should be tested more directly.

What's debt: the `_claims.py` attempt_number tracking for auto-steal is correct but the attempt number calculation in the acquire path is indirect — it derives from the existing claim row rather than tracking in events. This is fine for MVP but won't support FR-10 (escalation) cleanly when that lands.

## Working with the User

The user is technically precise and decisive. Two things stood out: (1) the schema-per-project proposal came with a clear rationale (connection multiplication, operational tax, federation cost) that was obviously correct once stated — a good sign that the user understands the system at the right level of abstraction, and (2) the user trusted the implementation approach and didn't micromanage, which let the build proceed efficiently.

The spec itself was exceptional — genuinely Level 3. The dependency hints, acceptance criteria, and invariant sections meant very few questions needed asking. The main ambiguity was whether claim operations generate events (they don't — claims are tracked in the claims table, not the event log).

## Friction and Improvements

- **psycopg3 pool configure callback**: The `configure` callback runs inside an implicit transaction. `conn.execute("SET synchronous_commit = on")` without `conn.commit()` leaves the connection in INTRANS state, which the pool rejects. Took two iterations to figure out. Better: psycopg-pool docs should call this out, or substrate should document the pattern.
- **dict_row with psycopg-pool**: Passing `row_factory=dict_row` to ConnectionPool isn't via a direct parameter — it goes through `kwargs={"row_factory": dict_row}`. Not obvious from the API surface. Several places in the code used tuple indexing (`row[0]`) before the dict_row migration was complete, causing KeyErrors at runtime.
- **Migration bootstrap chicken-and-egg**: `_substrate_migrations` table is created by the first migration, but `applied_versions()` queries it before any migration runs. Fixed by having the runner CREATE TABLE IF NOT EXISTS. Clean, but could be confusing for future implementers.
- **Replay false positives**: System events (custom_event, etc.) caused replay to halt because they're not in the workflow definition. Fixed by treating unknown transitions as no-ops. The spec doesn't address this case explicitly.

Suggested improvements:
- Add a `conftest.py` with proper fixture management (currently session-scoped substrate creates one project per test session, which means tests can interfere)
- The `_work_items.py` inline import of `_types` inside `create_work_item` should be a top-level import
- Consider adding a `_schema.py` or `_projection.py` module to separate projection update logic from event append logic

## Observations

The schema-per-project pivot was the single most impactful design decision of the session. It simplified three things simultaneously (connection management, operational burden, federation path) while giving up nothing. The original spec's DB-per-project choice was defensible in isolation but schema-per-project is strictly better for homelab scale. This is the kind of decision that's easy to miss at spec time because the costs (50 idle connections) only materialize when you're writing the connection pool code.

The spec's §17 (Concurrency Contract) and §18 (Projection Invariants) were the most valuable sections during implementation. Having the lock target, isolation level, event_seq allocator, and projection update protocol written down precisely meant the implementation could follow the spec mechanically rather than making concurrency decisions in the moment.

I'm uncertain about whether the replay is thorough enough. It replays transitions by looking up `to_state` in the workflow definition, but it doesn't re-derive custom_fields changes from transition payloads. For the current workflow definitions (which don't use payload-driven field updates), this is fine. But if a future workflow definition includes field mutations in transition payloads, the replay will need to be extended.
