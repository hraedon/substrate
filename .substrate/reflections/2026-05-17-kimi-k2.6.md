## On the project

The codebase is in a mature, well-tested state (561 tests passing). The security-audit breadcrumbs were mostly low-risk but real — the kind of operational hardening that matters in production. The spec/AGENTS.md convention is strong; the only friction is the "no comments" rule, which forces me to read the spec for intent rather than inline clues. That works because the spec is actually maintained.

BC-183 (further decomposition of `__init__.py` / `_in_memory.py`) is the real remaining structural issue. After this session, `__init__.py` is ~1393 lines and `_in_memory.py` is ~1619 lines. The Postgres side has extracted `_claims_api.py`, `_links_api.py`, `_recurrence_api.py`, `_transition.py`, but InMemory still embeds equivalent logic inline. This asymmetry will drift.

## On the work done

- **BC-172 (rfc8785 pin):** Trivial but correct. `==0.1.4` instead of `>=0.1.4`. No tests needed.
- **BC-173 (require_ssl):** The `_verify_ssl` helper checks `pg_stat_ssl` on every connection acquisition from the pool. This is slightly paranoid but matches the spec intent. Added `tests/test_connection_ssl.py` with 3 tests. One risk: if `pg_stat_ssl` is not readable by the connecting role, the check will raise. In practice the `substrate_test` user has access; in production this should be documented.
- **BC-178 (body size limit):** 10MB hard limit in the sole-signer middleware. FastAPI's `request.stream()` is unbounded by default, so this is a genuine DoS vector. The 413 response is tested with an 11MB payload. The limit is module-local (`max_body_size = 10 * 1024 * 1024`) and not configurable — acceptable for a sidecar default.
- **BC-179 (dead-letter atomicity):** The fallback path is a bit involved: when `events` row is missing (e.g., partition dropped), we parse `work_item_id` from `hook_queue.payload`, look up `work_items_current` for `workflow_name`/`workflow_version`, and only then append the `hook_dead_lettered` event. If the work item is also gone, we log a warning and skip the event. This preserves audit trail in the common case (partition drop is an admin operation, not concurrent traffic).
- **BC-180 (plaintext key warning):** One-line `log.warning` on every `KeySet._load()`. Accepted the spec position that in-memory plaintext is an environmental trust boundary. Environment-variable key injection was proposed but deferred — it would require broader `KeySet` refactoring.

Confidence is high on all except BC-179, where the fallback path has multiple branches but is hard to hit in a normal test (requires dropping a partition mid-dead-letter). The logic was verified by inspection.

## On what remains

1. **BC-183 — InMemory decomposition (next session priority):** Extract InMemory claims, links, hooks, and recurrence into standalone modules matching the Postgres extraction pattern. Target: both `__init__.py` and `_in_memory.py` ≤ 900 lines. This is medium-effort (~2 hours) and low risk because the contract layer (`_contract.py`) already owns the pure logic.
2. **Vendoring rfc8785 or cross-validation (BC-172 follow-up):** Pinned version buys time, but a vendored copy or secondary canonicalization implementation would eliminate supply-chain risk entirely. Low priority unless rfc8785 hits a CVE.
3. **Environment-variable HMAC key injection (BC-180 follow-up):** A `SUBSTRATE_HMAC_SECRET` env var path or direct value would help KMS-provisioned deployments. Needs `KeySet` constructor changes and test coverage.

## Gaps to flag

- `ConnectionManager._verify_ssl` queries `pg_stat_ssl` on every `connect()`/`transaction()` call. This adds one round-trip per connection acquisition. For high-throughput workloads, consider caching the SSL status on the connection object or making the check optional after first success. (`src/substrate/_connection.py:97–108`)
- `_hooks.py` `_move_to_dead_letter` has no `FOR UPDATE` lock on the work item. The spec says every mutation acquires `SELECT FOR UPDATE`, but the dead-letter path intentionally skips it because the work-item state is not modified. If this invariant changes, add the lock. (`src/substrate/_hooks.py:301–368`)
- Sidecar `max_body_size` is hardcoded to 10MB. For deployments with large event payloads, this may need to be configurable via `create_app(..., max_body_size=...)`. Currently not exposed. (`src/substrate/sidecar/app.py:30`)
- `__init__.py` docstrings were partially updated for `require_ssl` but `create_project` docstring still lacks the new parameter description. (`src/substrate/__init__.py:159–178`)
