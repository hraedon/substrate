# Session Reflection — 2026-05-17

**Work summary:** Resolved BC-183 by extracting InMemory backend methods into 7 new modules and Postgres orchestration into 3 new API modules. `_in_memory.py` reduced from 1,619 → 726 lines; `__init__.py` reduced from 1,393 → 1,231 lines. All 561 tests pass; lint clean.

## On the project

Substrate's codebase is well-structured with clear backend separation (Postgres vs InMemory) but had accumulated significant bloat in the two primary entry-point files. The Postgres backend already had extraction modules (`_transition.py`, `_claims_api.py`, `_links_api.py`, `_recurrence_api.py`), so the InMemory side was the obvious asymmetry. The extraction work confirms the existing pattern is sound: thin facade on the public class delegates to module-level orchestration functions that validate, time, and metric-wrap before calling the internal DB-side functions.

One concern: the `_in_memory.py` file still contains `register_workflow`, `register_workflow_file`, `get_workflow`, and a handful of private helpers (`_resolve_wf_def`, `_append_simple_event`). These are not large enough to justify extraction on their own, but the file is now at 726 lines — comfortably under the 900-line target.

## On the work done

The extraction follows the existing Postgres pattern closely: each thin facade method does minimal delegation with `from ._module import func` imports (lazy, no circular import risk). The new modules are:

**InMemory (7 modules):**
- `_in_memory_claims.py` — acquire/heartbeat/release/sweep + escalation check
- `_in_memory_links.py` — create/remove link
- `_in_memory_hooks.py` — poll hooks, dead-letter management
- `_in_memory_recurrence.py` — recurrence rule CRUD + fire
- `_in_memory_replay.py` — replay drift detection (~135 lines)
- `_in_memory_transition.py` — transition + `_validate_refs_in_memory`
- `_in_memory_work_items.py` — create/query/update_not_before + `_wi_to_work_item`
- `_in_memory_events.py` — append/read event

**Postgres (3 modules):**
- `_workflow_api.py` — register/register_file/get workflow
- `_work_items_api.py` — create/query/get work item
- `_events_api.py` — append/read event

All extractions preserve exact behavior: the property-based tests pass, which provides strong confidence since they exercise random sequences of operations on both backends.

## On what remains

1. **Phase B of BC-183 is complete.** The original breadcrumb mentioned extracting `create_work_item` and `append_event` for both backends — the Postgres versions are now in `_work_items_api.py` and `_events_api.py`, and the InMemory versions are in `_in_memory_work_items.py` and `_in_memory_events.py`.
2. The `_in_memory.py` file could go further with `register_workflow`/`get_workflow` extraction, but at 726 lines it's already well under the target.
3. `__init__.py` at 1,231 lines is still large (primarily docstrings on every public method). Further reduction would require extracting docstrings or shortening them, which is a style decision not a structural one.

## Gaps to flag

- **Lazy import pattern is inconsistent.** Some extracted modules use lazy `from ._contract import validate_mutation_params` inside functions; others import at top-level. The Postgres modules (`_workflow_api.py`, `_work_items_api.py`, `_events_api.py`) eagerly import `_Jsonb`, `ErrorCode`, etc., while the InMemory modules are mixed. A future cleanup should standardize on top-level imports in extracted modules since they no longer have circular-import risk.
- **BC-183 resolution note in breadcrumbs/resolved/183** is verbose but accurate. Future agent should be aware that Phase B is fully done, not partially.
- **`_work_items_api.py` returns `Event` as `object`** in its type annotation (`tuple[WorkItem, object]`) because of a typing mismatch between the public return type and the internal `_work_items.create_work_item` return type (`Event`). This is minor but worth noting at `src/substrate/_work_items_api.py:38`.
- **Metrics wiring in extracted modules** is hardcoded to `metrics.inc("workflows_registered", project)` etc. If metrics names change, there are now 3 extra files to update alongside `__init__.py`. Centralizing metric names in `_observability.py` as constants would reduce this risk.
