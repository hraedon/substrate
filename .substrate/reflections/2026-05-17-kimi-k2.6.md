# Session Reflection — 2026-05-17

**Work summary:** Resolved four open security-audit breadcrumbs (BC-174–BC-177), discovered and fixed two new gaps (BC-181, BC-182), extracted claim and link APIs from `Substrate.__init__.py` into `_claims_api.py` / `_links_api.py`, and filed BC-183 for the remaining decomposition work.

---

## On the project

Substrate is a genuinely impressive piece of engineering. The level of rigour — property-based conformance tests, cross-backend contract unity, explicit spec alignment — is rare in a Python library. The codebase is a pleasure to work in because conventions are enforced (frozen dataclasses, `Jsonb` wrapper, no comments, `_contract.py` as SSoT). The breadcrumb system is a strong institutional memory device; having resolved items in `breadcrumbs/resolved/` with full context means history is searchable and auditable.

The trade-off is growing file size. `__init__.py` at ~1,460 lines and `_in_memory.py` at ~1,620 lines are beyond the point where a reviewer can hold the whole file in working memory. The existing extractions (`_transition.py`, `_recurrence_api.py`) were done well and serve as templates, but the asymmetry between Postgres and InMemory backends (Postgres delegates to modules, InMemory embeds logic) is the real risk. That asymmetry was visible during this session — every change required touching `__init__.py` *and* `_in_memory.py` separately.

## On the work done

The user asked me to "scan the repo and pick out some breadcrumbs to resolve." I picked BC-174–BC-177 (the four lowest-severity open items from the security audit, plus BC-177 which was medium-severity but clearly bounded). All four were straightforward, well-scoped fixes. The test suite is fast enough that iterating on four changes plus full regression testing in one session is feasible.

While implementing BC-177 I noticed `register_actor_role` and `unregister_actor_role` were not covered by the `actor_id` validation — this was an honest gap, not a known bug. Fixing it immediately was the right call because the validation infrastructure was already in place; it was just a missing call site. Similarly, `TokenRegistry.from_file` had no defensive validation and would crash at sidecar startup with a raw Python `KeyError`. Both fixes were low-risk.

The decomposition work (`_claims_api.py`, `_links_api.py`) was motivated by the user's follow-up request to "take a swing at decomposing the big files." I extracted only the Postgres-side orchestration (timer + metrics + `_validate_mutation_params`), leaving the InMemory methods inline. This reduces `__init__.py` by ~70 lines — modest but meaningful. I did not attempt the heavier InMemory extraction because it would require pulling claim state updates out of `_in_memory.py` and into a new module, which is invasive and better scoped as a dedicated session. BC-183 captures this plan in detail.

One hesitation: `_claims_api.py` and `_links_api.py` accept `mgr` and `keys` as opaque objects, which is correct for the boundary, but the type signatures are weak (`object`). If typed, they'd need `from __future__ import annotations` plus forward-referenced dataclasses. The project already uses `from __future__ import annotations` everywhere, so this is a minor point, not a blocker.

## Gaps to flag

- **InMemory claim methods still inline** — `_in_memory.py:696-844` contain the full claim lifecycle. Any claim bug fix requires editing both `_claims.py` (Postgres) and `_in_memory.py`. This is the most likely source of future divergence.
- **Pydantic sidecar models don't enforce `max_length=255`** on `actor_id` — the server-side validation in `_contract.py` is sufficient for correctness, but the sidecar could reject earlier with a clearer HTTP 400. See `sidecar/models.py:70` and other `actor_id` fields.
- **`_in_memory.py` imports `validate_mutation_params` but also re-imports `validate_actor_id` inline** inside two methods. This is a local import to avoid circularity, but it's inconsistent with the rest of the file which imports contract functions at the top. `validate_mutation_params` was already imported at the top; adding `validate_actor_id` there should be safe and would clean this up.
- **Type annotations in `_claims_api.py` and `_links_api.py`** — the `mgr` and `keys` parameters are untyped. The project uses strong typing elsewhere; these modules are a minor regression.
- **Tests for `TokenRegistry` are in `tests/test_actor_id_and_token.py`** — a slightly odd home, but acceptable. If more sidecar auth tests are added, consider splitting to `tests/sidecar/test_auth.py`.
