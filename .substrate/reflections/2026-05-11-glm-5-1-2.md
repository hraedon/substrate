# Session Reflection — 2026-05-11

**Work summary:** Validated the full codebase (300 tests, lint clean), resolved 3 prior deferred breadcrumbs (BC-066/067/069), then conducted a 6-front adversarial review that identified 17 new issues. Filed all 17 as breadcrumbs (BC-071–087) and resolved 13 of them with code fixes. Final state: 395 tests, lint clean.

---

## On the project

Substrate is in remarkably good shape for its complexity. The RFC-062 contract extraction was a genuine quality win — having 20 pure functions in `_contract.py` meant that when I found bugs (zombie heartbeat, missing TTL validation), the fix was a 2-line addition to a pure function rather than spelunking through SQL-wrapped Postgres code. The property-based conformance tests caught real divergences between backends.

The biggest structural tension is the InMemory backend. It exists for testing but accumulates parity debt — my review found 5 divergences (read_events ordering, requeue work_item_id, validation order, missing JSON safety, replay from_state matching). Each is individually small but collectively they erode the conformance guarantee. The copy-on-write `_validators` fix was an example: `_hook_handlers` already used the safe pattern but `_validators` didn't, purely because nobody caught the inconsistency.

## On the work done

The session had two distinct phases: validation/resolution of prior work (clean, straightforward) and adversarial review (productive but intense).

The zombie heartbeat bug (BC-071) was the most satisfying find — `resolve_heartbeat` checked actor and attempt_number but never checked if the claim had expired. A misbehaving actor could hold a claim indefinitely by heartbeating past the TTL window, completely bypassing the steal/escalation mechanism. One-line fix in a pure function.

The replay fix (BC-072) was the most delicate. My initial fix was too aggressive — halting on all unrecognized transitions broke 5 tests because `append_event` intentionally allows custom transition names. The real bug was subtler: `custom_fields_update` was applied even for unrecognized transitions, potentially corrupting replayed state. The refined fix guards `custom_fields_update` behind `found=True`.

The resource leak fixes (BC-075) were straightforward but important — `create_project` and `__init__` could leak connection pools on failure. The `close()` idempotency fix (BC-085) was a natural companion.

I'm least confident about the heartbeat expiry check's interaction with the sweep path. The sweep deletes expired claims and emits events; now a heartbeat on an expired claim also raises CLAIM_LOST. If sweep and heartbeat race, both will fail — which is correct behavior, but worth noting.

## On what remains

Four deferred breadcrumbs need design decisions:

1. **BC-074 (high): `continue_on_revoked=True` skips signature verification** — The most impactful remaining item. The intent was to skip revocation status only, but the implementation skips the entire cryptographic check, including for completely unknown keys. Fix requires separating key lookup from status check.

2. **BC-079 (medium): Replay skips zero-event work items** — A work item in `work_items_current` with no events is the worst corruption case, and replay silently skips it. Should be treated as halted.

3. **BC-082 (medium): Default values not type-checked at registration** — A workflow with `type: integer, default: "high"` passes registration then fails every work item creation.

4. **BC-083 (medium): No uniqueness checks on names** — Duplicate state/transition/type names silently deduplicated via dict construction.

Additionally, BC-065 (hook savepoint deadlock), BC-068 (inconsistent workflow param types), and BC-070 (replay temp table accumulation) remain from prior sessions.

## Gaps to flag

- **`_signing.py` stored envelope trust**: `verify_event` with `stored_envelope` checks internal consistency (HMAC, hash) but never verifies the envelope matches the event fields. An attacker with write access to the events table could modify `actor_id` while leaving `canonical_envelope` untouched. Filed as part of BC-074's broader concern.

- **`_hooks.py:357-361` — consumer thread dies silently on initial connection failure**: After `start()`, the thread returns without any notification. `is_running` returns False but nothing alerts the application. Not filed as a separate breadcrumb — worth considering alongside BC-065.

- **`_claims.py sweep TOCTOU`**: `sweep_expired_claims` bulk-deletes expired claims before locking work items, creating a window where a concurrent acquire can create a phantom `claim_expired` event for an item already re-claimed. The UPDATE guard (`AND claimed_by = %s`) prevents projection corruption but the audit trail is misleading.

- **`_in_memory.py` validation gaps**: The InMemory backend still skips `validate_json_safe_value` in `_append_simple_event`, `_append_claim_event`, and `create_work_item`. These are internal paths not reachable from the public API, but they mean conformance tests can't catch null-byte injection through claim/link/escalation payloads. These are lower priority since the Postgres backend validates at `append_event`.

- **Test count grew from 300→395**: 95 new contract tests are good coverage but the 5 property-based conformance tests in `test_property_conformance.py` run with small hypothesis settings. Cranking up `max_examples` would catch more divergence cases at the cost of CI time.
