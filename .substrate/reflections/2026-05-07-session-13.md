## What went well

- The fresh scan caught a real correctness bug: `sweep_expired_claims` could clobber a newly-acquired claim under concurrent load. The fix was a one-line WHERE clause change (`AND claimed_by = %s` instead of `AND claimed_by IS NOT NULL`).
- The InMemorySubstrate divergences (M-2 through M-8) were systematic: ordering, event emission, type annotations, magic strings. All quick fixes.
- Using psycopg `SQL` composition instead of f-string for the claim-clear fragment eliminates a class of potential future issues.

## What surprised me

- The `sweep_expired_claims` race condition is the first real-backend correctness bug found since the early sessions. It would only manifest under genuine concurrent load (sweep running while a new claim is acquired on the same work item), which explains why the 270 tests never caught it.
- The `_check_escalation` return type divergence (`None` vs `bool`) in InMemorySubstrate was harmless because `acquire_claim` in InMemorySubstrate returns just `Claim` (not the full `tuple[Claim, bool, bool]`), so the escalation flag was never consumed. But fixing it to return `bool` is still correct for consistency.
