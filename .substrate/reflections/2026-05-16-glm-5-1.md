## On the project

Substrate is genuinely feature-complete for a coordination library. After 28 sessions and 160 resolved breadcrumbs, the codebase is remarkably well-exercised. The single-source-of-truth contract in `_contract.py` is the key architectural win — both backends call the same 20 pure functions, and the hypothesis property tests catch divergence that manual testing would miss.

The main tension I see is that `__init__.py` is still ~1400 lines even after extracting transition and recurrence. Each new feature adds facade methods. The extraction pattern (`_transition.py`, `_recurrence_api.py`) works but produces a lot of thin delegation boilerplate. At some point the `Substrate` class itself needs to become a composition of mixins or a pure facade, but that's a structural refactor with no functional benefit right now.

The sidecar implementation felt straightforward because the plan was unusually detailed. Plan 005 specified every endpoint, every error mapping, every test case. The implementation was mostly mechanical transcription. This is the best case for plan-driven development.

## On the work done

**What went well:**
- The transition extraction was clean. The `_transition.py` module follows the exact pattern of `_recurrence_api.py` (takes `mgr`, `keys`, `metrics`, etc. instead of `self`), making it a pure facade. All 511 core tests passed without modification, confirming the extraction was behavior-preserving.
- The sidecar came together in one pass. FastAPI's `extra="forbid"` on Pydantic models handles the sole-signer rejection... except it returns 422 instead of 400. I had to add a raw-body-checking middleware that runs before Pydantic validation. This is the right approach — intercept at the middleware layer, reject with the correct error code, never let the request reach the handler.
- The `_StderrLoggerFactory` approach for structlog in CLI mode avoids the pytest capture bug by resolving `sys.stderr` at print-time rather than configuration-time. This is a subtle issue that would have been hard to diagnose without seeing the CI failure pattern.

**What I'm less confident about:**
- The sidecar `auth_middleware` is registered inside `register_routes()` as a closure over the `tokens` parameter. This means it's a per-app middleware, not a proper ASGI middleware class. It works but isn't idiomatic FastAPI. If the sidecar grows more middleware (CORS, rate limiting), this pattern will get awkward.
- The sidecar tests don't exercise the hook claim/complete/fail lifecycle end-to-end because the test workflow doesn't declare hooks on its transitions. The `test_claim_complete_round_trip` and `test_hook_lease_expiry_requeues` tests are stubs (pass/skip). Plan 005 §10 calls for more thorough hook tests.

**What went wrong:**
- The structlog `PrintLoggerFactory(file=sys.stderr)` captured the stderr file object at configure-time, not at print-time. When pytest replaces `sys.stderr` with a capturing wrapper between tests, the old reference becomes invalid. Local runs passed because pytest's capture lifecycle is different when running a subset of tests. CI failed because it runs the full suite, which exercises the capture close/reopen cycle. Fixed with a custom factory that resolves `sys.stderr` lazily.
- The `.venv/bin/python` hardcoding in CLI integration tests was a pre-existing issue (from Session 27) that only manifested on CI because GitHub Actions uses a different virtualenv layout. Using `sys.executable` is the correct fix.

## Gaps to flag

- **Sidecar sole-signer middleware reads body before Pydantic** — `sole_signer_middleware` in `app.py:24-49` calls `request.body()` and parses JSON to check for `signature`/`payload_canonical_hash`. Starlette caches the body, so downstream handlers can re-read it, but this is an implicit contract. If a future Starlette version changes caching behavior, this breaks silently.
- **Sidecar `update_recurrence_rule` route** — `routes.py:345` tries to read `rule_id` from `request.query_params`, but the route is `POST /v1/update_recurrence_rule` with no path parameter. The `rule_id` should come from the request body. This is a bug — it will raise a `KeyError` at runtime.
- **Sidecar auth middleware ordering** — the auth middleware is registered inside `register_routes()` via `@app.middleware("http")`, which means it runs in LIFO order relative to other middleware. The sole-signer middleware in `create_app()` runs first (outermost), then auth runs second (innermost). This is correct but fragile — adding middleware in the wrong registration point would silently break auth.
- **No sidecar conftest.py** — `tests/sidecar/test_sidecar.py` uses module-scoped fixtures that create and destroy Postgres schemas. If sidecar tests grow beyond one file, the fixtures should move to `tests/sidecar/conftest.py`.
