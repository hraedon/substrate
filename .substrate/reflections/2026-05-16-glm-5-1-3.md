# Session Reflection — 2026-05-16

**Work summary:** Closed 9 breadcrumbs (BC-161 through BC-169), all in the sidecar and recurrence subsystems. Started from user-provided notes on BC-161–164, fixed those, then ran a full audit that caught 5 more issues including two critical bugs (heartbeat model mismatch, recurrence count exhaustion). 529 tests pass, lint clean.

---

## On the project

The sidecar layer (Plan 005) has a characteristic pattern: the core library is solid, but the HTTP thin layer on top accumulated parameter mismatches, dropped fields, and raw-dict routes. The core API surface is well-designed with good invariants; the sidecar was clearly written quickly to match it and missed several details. The recurrence subsystem has a genuine correctness bug in count exhaustion that affects both backends identically — suggesting the InMemory was ported line-by-line from Postgres, bugs included.

The test suite is strong on the core library (511 tests) but thin on the sidecar (18 tests, up from 15 this session). The sidecar tests exercise happy paths well but skip error cases, edge cases, and most individual routes. The hook lifecycle tests were stubs until this session.

## On the work done

Confident in all 9 fixes. The recurrence count bug (BC-166) was the most consequential — any user with a finite-count recurrence rule would see it fire forever after exhausting. The claim_hooks stranding fix (BC-167) is clean: only mark rows in_progress that we'll actually return.

The BC-164 fix (closed-form interval arithmetic) is a clear improvement but I'd want a second look at the edge case where `after_slot` equals `now` exactly (elapsed = 0, steps = 1, candidate = after_slot + interval — correct).

The sole-signer middleware fix (BC-162) uses `request._body` which is technically a private attribute, but it's the same attribute Starlette uses internally, so this is making the implicit contract explicit rather than introducing a new dependency.

## On what remains

From the audit, lower-severity items I didn't fix:

- **InMemory missing claim_hooks/complete_hook/fail_hook/sweep_expired_hook_leases** — the external hook processing API isn't implemented in InMemorySubstrate. Only affects consumers trying to use the sidecar-like pattern against InMemory for testing.
- **Sidecar test coverage** — still missing tests for ~15 routes (update_not_before, create_link, remove_link, replay, recurrence CRUD, dead-letter, read_events_since, heartbeat with expected_attempt_number, custom_field_filters in query).
- **register_recurrence_rule sets next_fire = start_at when compute_next_fire returns None** — wastes one fire_recurrence cycle for rules where start_at >= end_at before marking exhausted.

## Gaps to flag

- `_in_memory.py:1375-1396` — the count_remaining fix was applied but the InMemory recurrence fire_recurrence hasn't been independently tested with count-limited rules; the test suite uses InMemory for recurrence tests but the count exhaustion path may not be covered.
- `src/substrate/sidecar/routes.py` — no route-level test for `heartbeat_claim` with `expected_attempt_number` now that it's wired through.
- `src/substrate/_hooks.py:148-153` — the claim_hooks fix skips rows without work_item_id, but those rows remain `pending` in the queue. The only path to dead-letter them is the internal hook consumer (`poll_and_process_hooks`), which external sidecar consumers bypass. A hook enqueued by a future code change that omits work_item_id from the payload would sit pending forever when using the sidecar.
- `_find_next_future_slot` has no unit test for the new closed-form interval path with negative or zero-duration intervals (guarded by `<= 0` check but untested).
