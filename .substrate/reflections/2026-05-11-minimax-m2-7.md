# Session Reflection — 2026-05-11

**Work summary:** Full adversarial review of the substrate codebase from an attacker perspective. Read all major source files (18 files in src/substrate/), the full spec, migrations, and test structure. Identified 12 security/correctness issues spanning 4 critical, 3 high, and 5 medium severity. Created breadcrumb entries for all 12 in `breadcrumbs/pending/`, updated the index with 6 high/critical items, and updated the pending README with all 12.

---

## On the project

Substrate is a well-architected event-sourced coordination library with genuine strengths: RFC 8785 canonical signing, gap-free `event_seq` allocation, dual-backend conformance testing via `_contract.py`, and a disciplined migration system. The spec is thorough and the breadcrumb culture shows active maintenance.

The adversarial lens reveals a pattern: **the system is well-hardened against accidental misuse but less so against intentional abuse**. Rate limiting, input validation on `event_id` UUID format, and unbounded `not_before` are notable gaps. The `actor_metadata` trust tier (actor-claimed, not independently verified) is acknowledged in the spec but the implications for self-registering actors may not be obvious.

The most significant architectural concern is #101: actors can register their own roles in `actor_roles` and then claim those roles in `actor_metadata`, passing FR-24 enforcement. The system provides a signed audit trail but no out-of-band verification. This is by design per BR-09, but it's a trust model that should be explicitly evaluated against deployment threat models.

## On the work done

Confident in the identification of all 12 issues. The critical ones (#100, #101, #102, #103) are real and exploitable in adversarial deployments. The high ones (#104, #105, #106) are load-bearing spec properties that aren't fully delivered.

**Self-corrected during review**: Initially thought #108 (`synchronous_commit` per-session) was a bug, but after deeper analysis the psycopg pool's `configure` callback runs on every connection checkout and raises on failure, so bad connections don't enter the pool. The per-session setting is actually the spec's stated intent. Left #108 and #109 in pending as medium severity for documentation purposes but flagged them as likely false positives in the entries.

**One open question**: #105 (revoked-key events left in log with `continue_on_revoked=True`) interacts with deferred item #074. Both need resolution together since they address the same feature from different angles.

## On what remains

**Immediate (should be addressed before production multi-tenant deployment):**
1. Add rate limiting to `acquire_claim`, `transition`, `append_event`, and link operations
2. Validate `event_id` as proper UUIDv4 and reject NIL UUID
3. Add maximum bound to `not_before` delta (e.g., 1 year max)
4. Add `expected_event_seq` to `create_link` and `remove_link`
5. Evaluate whether actors should be allowed to register their own roles, or if that requires an admin gate

**Design decisions needed:**
- #105: Should `continue_on_revoked=True` record which events were skipped, rather than just counting warnings?
- #110: Is shallow merge of `custom_fields` intentional? If so, document it. If not, implement deep merge.
- #111: Audit `_workflow_schema.json` for `additionalProperties: true` and determine which extra fields are intentionally permitted.

**Nice to have:**
- Maximum payload size limit (64KB suggestion)
- Connection pool circuit breaker to reject with `SERVICE_UNAVAILABLE` when saturated
- Per-actor request quotas

## Gaps to flag

- **No payload size limits anywhere** — `append_event` and `append_transition_event` accept unbounded `payload: dict`. A malicious actor can attach multi-MB JSON. No defense at API boundary. (`_events.py:97`, `__init__.py:443`)
- **`create_link` and `remove_link` lack `expected_event_seq`** — these are the only event-producing operations without optimistic concurrency control. TOCTOU race documented in #104. (`_links.py:44`, `_links.py:125`)
- **`actor_metadata.role` is not independently verified** — actors who can call `register_actor_role` for themselves can pass any role gate. The FR-24 enforcement is only as strong as the `actor_roles` registration process. (`_actor_roles.py:71-83`, `_contract.py:76-89`)
- **`validate_work_item_refs` at line 325** raises unhandled `ValueError` from `uuid.UUID(value)` — implementation detail leak. `_coerce_field` at line 407-414 handles this correctly with try/except. Inconsistent within the same file.
- **`update_not_before` has no maximum bound** — any future datetime accepted. Can permanently DOS a work item. (`__init__.py:1175-1253`, `_contract.py:51-56`)
- **Hook consumer relies on LISTEN/NOTIFY as latency optimization** — if NOTIFY is dropped (buffer max 8 entries), hooks only processed on 30s poll. For a system claiming p99 < 1s dispatch, this is a gap. Spec acknowledges polling as correctness floor but doesn't address NOTIFY drop scenario explicitly.
- **Breadcrumb 074 (deferred) and 105 interact**: #074 is about skipping signature verification entirely; #105 is about what happens to the skipped events. Both need resolution for FR-25 to be safe to use.
