## On the project

Substrate is in exceptional shape for a pre-1.0 project. The 182-resolved breadcrumb backlog tells the story better than anything else — every issue found by adversarial reviewers has been systematically tracked and closed. The spec is authoritative and the code follows it. The dual-backend architecture (Postgres + InMemory) with `_contract.py` as the shared validation layer is genuinely well-executed.

The most striking observation is the institutional memory. Between `AGENTS.md`, `breadcrumbs/README.md`, `.substrate/worklog.md`, and the reflection files, a new agent can orient to the full project context in minutes. Most mature production codebases have worse onboarding documentation than this pre-release library.

The three concerns I raised (API surface, trust model, operational complexity) are all real, but they're also all acknowledged. The spec's §20 consumer expectation boundary is unusually honest about what substrate doesn't do. The debate/ directory and resolved breadcrumbs show these issues have been thought about before. The contribution of this session is turning "acknowledged gap" into "concrete proposal with implementation path."

## On the work done

The code hygiene fixes were straightforward and low-risk:

1. `_work_items_api.py:27` — `tuple[WorkItem, object]` → `tuple[WorkItem, Event]`. The reflection from 2026-05-17 flagged this as "minor but worth noting." It was exactly that — a 2-line fix that improves type safety for downstream consumers.

2. Lazy import hoisting across 6 InMemory modules. The extracted modules (`_in_memory_transition.py`, `_in_memory_claims.py`, etc.) had imports lazily placed inside function bodies, a remnant of the extraction session (BC-183). Hoisting them to module level makes the dependency graph explicit and removes the redundant `import uuid` in `_in_memory_transition.py:155`. Only two genuinely circular imports remain lazy (`_in_memory_hooks → _in_memory_claims` and `_in_memory_work_items → _in_memory_transition`).

3. Import consolidation in `_events_api.py` — 5 separate `from ._contract import` blocks merged into one.

The three RFCs (Plans 007–009) are the main deliverable. They follow the existing plan format and reference specific spec sections, breadcrumbs, and FR numbers. Each presents multiple design options with a recommended path. Plan 008 (trust hardening) is the most immediately actionable — WS-1 (strict roles) and WS-5 (key rotation safety) are small pure-contract changes. Plan 009 (operational runtime) is the most architecturally significant — it revises the "library, not daemon" stance that's been load-bearing since session 1.
