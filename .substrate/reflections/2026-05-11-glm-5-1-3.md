# Session Reflection — 2026-05-11

**Work summary:** Implemented three operational-hardening mitigations for sync validator row-lock DoS (BC-112): Postgres `statement_timeout`, AST-level I/O detection at registration time, and near-timeout watchdog with structured logging and Prometheus metric. Filed breadcrumb, resolved it, wrote 13 tests. All 401 tests pass, lint clean.

## On the project

Substrate is in remarkably good shape for a project that's been through multiple adversarial review rounds. The codebase is clean, well-structured, and the convention of frozen dataclasses + dict_row factory + SQL composition via psycopg is consistent throughout. The `_contract.py` single-source-of-truth pattern is genuinely good — pure validation functions that both backends delegate to eliminates an entire class of divergence bugs.

The no-comments convention works better than I expected. Function names and structure carry the intent. The spec + AGENTS.md are the real documentation, and the code trusts you to read them.

## On the work done

The three mitigations were straightforward individually, but the AST-level I/O detection required careful thought about what "best-effort" means in practice. The key insight: checking `handler.__globals__` for names that appear in the handler's AST catches the common case (module-level imports used in the handler) while gracefully handling edge cases (lambdas, builtins, closures) by allowing with a warning. This is the right trade-off for an operational-hardening measure in a single-tenant system.

The `statement_timeout` is a one-liner defense-in-depth — it doesn't change behavior today because validators run in a ThreadPoolExecutor thread that doesn't touch the connection, but it bounds the transaction's SQL execution window at the Postgres level.

The watchdog (80% threshold, structured log + metric) is simple and gives operators signal before a validator actually times out. The metric name `validators_near_timeout` follows the existing naming convention.

I'm confident in all three. The I/O detection is best-effort by design — it won't catch dynamic imports or closures over I/O modules, and it shouldn't try to. The test coverage is solid: 5 I/O detection tests (reject socket, psycopg, subprocess; allow pure, lambda, builtin), 2 watchdog tests, 3 statement-timeout integration tests, 2 API integration tests.

## Gaps to flag

- `check_validator_io_safety` inspects `__globals__` but not closure variables (`__closure__`). A handler that captures an I/O-bound object via closure won't be caught. Acceptable for best-effort, but worth documenting if operators ask about false negatives.
- The `statement_timeout = '5s'` is hardcoded to match the Python timeout. If someone changes `run_validator`'s default timeout, they need to update the SQL too. Could be parameterized, but the coupling is intentional — both timeouts should agree.
- `test_contract.py` and `test_phase3.py` have pre-existing lint issues (unused imports, timezone.utc vs UTC, mutable class attributes). Not introduced by this session but noted during the lint sweep.
