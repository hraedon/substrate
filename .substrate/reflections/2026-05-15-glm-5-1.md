# Session Reflection — 2026-05-15

**Work summary:** Implemented all 16 open breadcrumbs (BC-145 through BC-160) in severity order: 3 critical, 5 high, 6 medium, 2 low. All 480 tests pass, lint is clean (including the pre-existing UP037 in `_in_memory.py`). Breadcrumbs moved to resolved, README index updated.

## On the project

The project is in strong shape. The breadcrumb backlog is now zero — every registered defect has been addressed or explicitly accepted. The codebase is well-organized: the `_contract.py` single-source-of-truth pattern, the event-sourcing + projection hybrid, and the schema-per-project isolation are all coherent. The testing surface is broad (480 tests) and the conformance tests between Postgres and InMemory backends catch real divergence.

The main structural tension is the partitioned events table (migration 010) — it forces compromises on global uniqueness (BC-148, addressed via advisory locks) and partition pruning (BC-153, partially addressed with upper-bound filtering). A `created_at` column on `work_items_current` would enable full lower-bound pruning but requires a migration.

## On the work done

Went well: The severity-ordered approach was efficient. Critical BCs (146, 148, 149) were all real correctness bugs with clear fixes. The advisory lock for BC-148 was the right call — lightweight, transaction-scoped, no schema changes needed. The replay orphan check (BC-145) ended up restructuring the event-fetching loop which was a nice side improvement.

Awkward: BC-157 (payload equality in idempotency) was a trap. I initially added strict payload comparison to `check_idempotency`, which broke existing tests because events like `link_created` embed a randomly-generated `link_id` in the payload. Reverted to accepting the original behavior — the BC-150 UniqueViolation recovery path provides the real protection. The lesson: payloads in this system are not deterministic; idempotency must be keyed on structural fields (actor_id, transition, work_item_id), not payload content.

The BC-153 fix (partition pruning) is partial — only an upper bound. The `read_events_by_work_item` queries now use `last_event_at` to prune future partitions, but without a `created_at` column there's no lower bound to prune older ones. This is documented as accepted; at homelab scale it's unlikely to matter.

## On what remains

No open breadcrumbs remain. The project is in a clean state.

Medium-term improvements that could be filed as future breadcrumbs:
- Add `created_at` column to `work_items_current` for full partition pruning (BC-153 follow-up)
- The `_in_memory.py` backend doesn't have the advisory lock equivalent for BC-148 (single-threaded, so it doesn't need it, but worth noting for completeness)
- CLI integration tests are still noted as "not yet added" in AGENTS.md (Plan 002)

## Gaps to flag

- `_replay.py` orphan check queries all events in one pass (`SELECT ... FROM events ORDER BY work_item_id, event_seq`) instead of per-work-item. For very large event tables this could be a memory concern. Current code pre-fetches into a dict grouped by work_item_id. Acceptable at homelab scale but would need streaming for production-scale deployments.
- `_events.py:read_events_by_work_item` now does an extra `SELECT last_event_at FROM work_items_current` on every call. This is a single-row PK lookup so it's fast, but it's a new query that wasn't there before.
- `_integrity.py:_parse_semver` now raises `SubstrateError` instead of a bare exception — this changes the error handling contract for the caller. The caller's `except Exception` catch was removed. If `_parse_semver` is called from anywhere else, that code needs updating too (currently only called from `check_integrity`).
