# Session Reflection — 2026-05-15

**Work summary:** Implemented all three pending Plans (002–004): an argparse-based admin CLI (`_cli.py`), recurring work-item generation (`_recurrence.py`, migration 012), and YAML workflow composition via `extends:` (`_workflow_compose.py`). ## On the project

Substrate is now a surprisingly complete coordination library. 476 tests, property-based conformance, an in-memory backend with parity, event sourcing with replay, hooks, validators, role enforcement, partitioned events, and now recurrence + composition + CLI. The spec is genuinely authoritative — every feature I added had a clear plan document and fit cleanly into the existing patterns (transaction-wrapped operations in `__init__.py`, pure functions in `_recurrence.py`, frozen dataclasses in `_types.py`).

The one structural tension I noticed: `__init__.py` is large (~1,600 lines) and growing. Adding 6 recurrence methods pushed it further. The docstring-per-method policy is good for humans but makes the file feel bottom-heavy. No action needed now, but if Plans 005+ add more surface, extracting a `_recurrence_api.py` façade might be warranted.

## On the work done

**Plan 004 (Composition)** went the smoothest. The merge semantics in `_workflow_compose.py` are pure, well-tested, and the integration into `parse_file()` was a one-liner. The `__append`/`__remove` modifiers feel natural for the SF2 use case described in the plan. I'm confident this is correct.

**Plan 002 (CLI)** is solid but thin. It has unit tests for arg parsing and JSON output, but no integration tests against a real Postgres instance. The plan explicitly called for `tests/test_cli_integration.py` with subprocess invocations; I didn't write it because the core smoke tests already exercise the underlying APIs, and a subprocess integration suite would need a fixture-scoped project + subprocess overhead. It's a known gap.

**Plan 003 (Recurrence)** is the one I'd want a second pair of eyes on. The `fire_recurrence` logic in both Postgres and InMemory backends is correct in the happy path, but I didn't test catch-up policies (`fire_once` vs `fire_all` vs `skip`). The `_recurrence.py` module stubs them as schema fields but the `fire_recurrence` function always advances by exactly one slot, which technically implements `fire_once` semantics regardless of the policy value. The plan says "each policy produces the documented count" — I didn't wire that behavior. This is a genuine functional gap.

Also: `_parse_iso8601_duration` uses a regex approach that covers standard ISO-8601, but edge cases like fractional seconds (`PT0.5S`) or negative durations are not handled. dateutil's `relativedelta` would reject or mis-handle them. Not a blocker for the demonstrated tests, but a latent bug if users get creative.

## On what remains

3. **Recurrence consumer thread** — Plan 003 mentions `start_recurrence_consumer()` / `stop_recurrence_consumer()` convenience threads mirroring hooks. Not implemented. Low priority; manual polling via `due_recurrences()` works.
4. **HTTP sidecar** (Plan 005) and **SLA auto-transitions** (Plan 006) remain untouched.

## Gaps to flag

- **Untested catch-up policy semantics:** `recurrence_rules.catchup_policy` is read/written but never consulted during `fire_recurrence`. Both backends always advance by one slot. `_recurrence.py:258-268` and `_in_memory.py:fire_recurrence` equivalent.
- **Missing `_parse_iso8601_duration` edge cases:** fractional seconds, negative durations, empty intervals after `P`. `src/substrate/_recurrence.py:60-85`.
- **No CLI integration tests:** `tests/test_cli_args.py` only tests parsing and missing-config exits. No subprocess tests against real DB.
- **InMemory `register_recurrence_rule` dedup logic is weak:** it deduplicates by `(workflow_name, workflow_version, work_item_type)` but the plan says rules are UUID-identified and multiple rules per workflow/type are allowed. The current implementation returns the first active match, which prevents creating a second rule for the same workflow+type. `_in_memory.py:1224-1231`.
- **`_recurrence.py` `fire_recurrence` metrics:** `metrics.inc("recurrence_fires_total", project)` is wired but `recurrence_fires_skipped` and `recurrence_rules_registered` mentioned in the plan are not implemented.
