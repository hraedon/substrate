# Session Reflection — 2026-05-08

**Work summary:** Resolved five adversarial-review breadcrumbs (BC-055 through BC-059) and promoted two draft RFCs to numbered items (BC-060, BC-061). All 282 tests pass. Lint clean.

## On the project

The adversarial-review breadcrumbs were well-written — clear reproduction steps, accurate line references, and explicit impact statements. BC-055 (the TOCTOU in `update_not_before`) was the only high-severity item and the most interesting: in both backends, the projection mutation happened before the idempotency check inside `_append_event`. The fix was straightforward (check early, return early), but the bug shows how subtle TOCTOU can be in event-sourced systems — you need to reason carefully about every mutation path and whether it's inside or outside the idempotency guard.

BC-058 (actor_kind misattribution in claim events) is a good example of a pattern I've seen in the codebase before: the `_append_claim_event` helper in the InMemory backend was hardcoded to `actor_kind="system"`, and the Postgres `acquire_claim`/`release_claim` had the same. The fix required threading `actor_kind` through the public API, internal functions, and the helper. The public API change is backward-compatible (default `"agent"`), but it's an API surface expansion that downstream consumers should be aware of.

## On the work done

BC-055: Moved `check_idempotency` call before the `UPDATE work_items_current SET not_before` in the Postgres backend. In the InMemory backend, added an early idempotency check before mutating `wi["not_before"]`. Both paths now match the pattern used in `_append_event` — check first, mutate only if new.

BC-056: Added `attempt_number: int = 0` to the `WorkItem` dataclass. The SQL already fetched it, the dict stored it, but the dataclass silently discarded it. Backward-compatible additive change.

BC-057: Set three live-snapshot columns (`last_event_at`, `claimed_by`, `claim_expires_at`) to NULL in the replay output table INSERT. The `_states_match` function was already correct (only comparing replayed fields), but the output table was semantically confusing. NULL makes it self-documenting.

BC-058: Added `actor_kind` parameter (default `"agent"`) to `acquire_claim` and `release_claim` in both backends and the public `Substrate` API. Claim events now use the caller's `actor_kind` instead of hardcoded `"system"`. `sweep_expired_claims` still uses `"system"`, which is correct.

BC-059: Made `validate_field_update`, `_rebuild_wf`, and `CustomFieldDef.from_dict` accept both `"default_value"` (DB key) and `"default"` (YAML key) via `.get("default_value", .get("default"))` fallback. Prevents a silent regression if anyone normalizes DB storage to use the JSON Schema key.

## Gaps to flag

- **BC-055 InMemory parity concern:** The InMemory `update_not_before` now has its own idempotency check inline (checking `_event_id_index`), which is structurally different from the Postgres path that calls `_check_idem` / `check_idempotency`. This works, but the code could benefit from a shared `_check_event_id_not_used(event_id, actor_id, transition)` helper that both backends call consistently.
- **BC-058 `sweep_expired_claims` in InMemory:** The `sweep_expired_claims` method still calls `_append_claim_event` without `actor_kind`, which defaults to `"system"`. This matches the Postgres behavior. But the `_append_claim_event` signature now has `actor_kind` as a kwarg with default `"system"` — this is correct but could be confusing. The escalation path also uses the default, which is correct.
- **No migration needed for BC-056:** The `WorkItem` dataclass gained an `attempt_number` field, but the column already existed in the DB (it was added in the BC-054 fix). The InMemory backend already stored it in the dict. So no migration — just a code-level fix.
