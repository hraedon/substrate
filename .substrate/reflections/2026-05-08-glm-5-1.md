# Session Reflection — 2026-05-08

**Work summary:** Resolved BC-054 by persisting `attempt_number` on `work_items_current` (Postgres) and `_work_items` dict (InMemory) instead of deriving it from the transient claims table/dict. Both backends had the same bug: after claim release/transition/sweep, the claim row/entry is deleted, causing the next `acquire_claim` to start at attempt_number=1 regardless of how many times the work item had been claimed before. Added migration 006, updated replay drift detection, fixed test assertions.

---

## On the project

The codebase is well-organized and the spec-first approach means intent is usually discoverable. The dual-backend pattern (Postgres + InMemory) is a real strength for testing, but it creates a recurring parity risk — every feature must be implemented twice, and the InMemory backend is easy to miss edges on. The breadcrumb system is good at catching these parity gaps, though the BC-054 writeup incorrectly assumed only InMemory was affected. The migration system is clean and auto-discovered, which made adding 006 straightforward.

## On the work done

The investigation was the hardest part. BC-054 assumed InMemorySubstrate had the bug and Postgres was fine. Reading the Postgres `_claims.py` revealed both had the identical bug: `DELETE FROM claims` on release/transition, then `attempt_number = 1` on next acquire because no existing claim row is found. The fix was clean — store `attempt_number` on the work item row, increment on every `acquire_claim`, never reset it on claim removal.

One thing I'd want a second pair of eyes on: the `test_e2e.py` assertion changes. The old test asserted `attempt_number == 1` on the second and third claims of a work item, which was the buggy behavior. I changed them to `2` and `3` respectively, which is correct under the new semantics. But this is a semantic change — `attempt_number` now means "how many times has this work item been claimed" not "how many times has this claim been acquired".

## Gaps to flag

- **`check_migrations_current` only checks max version** (`_migrations.py:85`): It checks `if max_available not in applied`, which means deleting a middle migration (e.g., version 4 out of 6) won't trigger `MIGRATION_REQUIRED` as long as version 6 is in the applied set. The test had to be updated to delete version 6 (the latest) for it to fail. This is a real gap — a partially-applied migration set with the latest version present would pass the check but have a broken schema.
- **Replay CREATE TABLE uses `SELECT * FROM work_items_current WHERE 1=0`** (`_replay.py:54`): This automatically picks up new columns, which is good for forward compat, but it means the replay table schema depends on the live table schema at replay time. Not a bug, but worth noting.
- **`_work_items.py _row_to_work_item` doesn't include `attempt_number`**: The `_WORK_ITEM_FIELDS` SELECT now includes it, but the `WorkItem` dataclass doesn't have that field, so `_row_to_work_item` silently ignores it. This is fine — `attempt_number` is internal, accessed via `Claim.attempt_number` — but it means the extra column is fetched and discarded on every `query_work_items` call. Minor, not a problem.
