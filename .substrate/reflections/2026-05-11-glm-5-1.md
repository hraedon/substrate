# Session Reflection — 2026-05-11

**Work summary:** Triaged and resolved all 17 open/deferred Minimax breadcrumbs (BC-100–111, BC-065–083). Fixed 7 real bugs (BC-103, 106, 107, 074, plus 4 found during codebase scan). Resolved 10 as false alarms or by-design. Updated spec to reflect schema-per-project isolation model.

---

## On the project

The adversarial review process is working as intended — it surfaced real issues alongside noise. The signal-to-noise ratio was about 40% (7 real bugs out of 17 breadcrumbs). The remaining 60% fell into predictable categories: environmental trust boundaries (plaintext memory, self-attested roles), scope mismatches (rate limiting for a library), and confused analysis (the synchronous_commit breadcrumb was literally the reviewer thinking out loud and forgetting to delete it).

The most interesting finding was how InMemory replay had quietly diverged from Postgres replay in two significant ways — skipping entire events on revoked keys, and applying custom_fields unconditionally. These undermine the conformance testing goal of RFC-062. The property-based conformance tests caught a symptom (the pre-existing `test_replay_equivalence` failure) but didn't isolate the cause. The InMemory backend is always going to be a maintenance burden; the single-source-of-truth contract in `_contract.py` helps but doesn't cover replay logic.

The spec update to schema-per-project was overdue. The code has never done DB-per-project, and the breadcrumbs/worklog/reflections all document the pivot happening during the MVP build session. The spec was the only document still describing the original design. The v4 changelog entry documents the amendment clearly.

## On the work done

Confident in all fixes. The trickiest was BC-074 (continue_on_revoked signature verification) because it required understanding the key lifecycle in both backends and matching their behavior. The datetime timezone issue in BC-106 (naive vs aware datetime comparison) was a classic Python footgun caught by tests immediately.

The 4 additional bugs found during the codebase scan were more satisfying than the breadcrumb work — they were genuine oversights from prior sessions (InMemory replay divergences, missing `work_item_id` in idempotency check, `{}` stored as NULL). These are the kind of thing that compound silently.

Less confident about not filing breadcrumbs for the low-severity items (close() exception safety, duplicated check_expected_seq, JSON validation ordering). They're genuinely low priority but the convention is to file what you notice. I chose not to because they'd add noise without action.

## On what remains

The pre-existing `test_replay_equivalence` property test failure should be investigated. It fails deterministically with the sequence `create → claim → append_event(transition=None)`, producing drift=0 on Postgres and drift=1 on InMemory. This likely relates to the replay divergence fixes I just made, but it was failing before my changes too, so something else is going on in the InMemory replay path.

The `_events.py` and `_contract.py` have a duplicated `check_expected_seq` function. Should consolidate — `_events.py` should import from `_contract.py`.

The `close()` method should wrap `_hook_consumer.stop()` in try/finally to ensure `_mgr.close()` always runs.

## Gaps to flag

- `tests/test_property_conformance.py::test_replay_equivalence` — pre-existing deterministic failure. Not flaky; reproduces every run with the same minimal sequence. Likely an InMemory replay logic gap in the append_event(transition=None) path.
- `_events.py:76` vs `_contract.py:184` — identical `check_expected_seq` function in two modules. Maintenance hazard.
- `_events.py:112-127` — JSON safety validation runs AFTER idempotency check in Postgres, BEFORE in InMemory. Divergent behavior for duplicate event_id with invalid payload.
