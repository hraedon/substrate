# Session Reflection — 2026-05-11

**Work summary:** Implemented RFC-062 — the single-source-of-truth backend contract. Created `_contract.py` with 20 pure validation/decision functions, refactored both `_in_memory.py` and the Postgres backend (`__init__.py`, `_claims.py`, `_links.py`, `_actor_roles.py`) to delegate to it, and added 5 property-based conformance tests using hypothesis. Net result: -347 lines of duplicated logic replaced by one shared module.

---

## On the project

Substrate is in an unusually mature state for its scale. The dual-backend architecture (Postgres + InMemory) was always going to be the biggest maintenance tax, and RFC-062 was the right structural fix. The fact that 10 InMemory parity bugs were found across sessions 13-17 confirms the diagnosis — hand-maintained parallel implementations will always drift.

The codebase is clean and well-organized. The frozen dataclass pattern for domain types, the schema-per-project isolation, and the event-sourced + projection hybrid are all well-executed. The spec-first discipline (spec.md, spec.yaml, AGENTS.md) is unusually strong for a project this size.

One thing I noticed: the `__init__.py` at 1358 lines is a pure façade, but the import structure is unwieldy — every `_types` export is individually aliased. This is cosmetic but makes the file harder to scan.

## On the work done

The contract extraction went smoothly. The key insight was to focus on *pure decision functions* — functions that take state as input and return decisions, with no storage side effects. This cleanly separates "what should happen" (contract) from "how to persist it" (backend).

The claim lifecycle extraction (`resolve_claim_acquire`) was the most complex piece — it replaced ~70 lines of duplicated claim decision logic in each backend with a single 60-line pure function. The Postgres version was particularly satisfying because it let me replace the INSERT/UPDATE branching with a single `INSERT ... ON CONFLICT DO UPDATE` UPSERT.

The property-based tests are a good start but are limited by the sequential, single-threaded nature of InMemory. They can't catch concurrent interleaving bugs (the `SELECT FOR UPDATE` vs dict-lock difference). They *can* catch logical divergence, which is the more common class.

What I'm most confident in: the pure validation functions (actor_kind, role_gating, idempotency, expected_seq, link_type, read_events_filters). These are straightforward and well-tested by the existing 295 tests.

What I'd want second eyes on: `resolve_claim_acquire` — the claim acquisition logic is the most nuanced piece, and while all 300 tests pass, the edge cases around TTL boundaries and claim expiry timing deserve careful review.

## On what remains

Nothing actionable remains. All breadcrumbs are resolved, all RFCs are resolved. The project is in a "done" state.

Potential future work (not urgent):

## Gaps to flag

- `tests/test_property_conformance.py` — the hypothesis tests are marked `@pytest.mark.slow` and won't run in the default `pytest` invocation. They should be added to the CI `slow` job (if one exists) or run nightly.
- The `_contract.py` module is new and has no direct unit tests — its correctness is validated implicitly through the existing 295 tests that exercise both backends. Direct contract-level unit tests would be valuable for faster iteration.
- `src/substrate/_claims.py:89-96` — the UPSERT pattern (`INSERT ... ON CONFLICT DO UPDATE`) replaced the previous INSERT-or-UPDATE branching. Functionally equivalent, but the original code had separate code paths for "no existing claim" vs "expired claim". The UPSERT is cleaner but worth verifying against the migration schema to ensure the unique constraint on `work_item_id` exists (it does — migration 001).
