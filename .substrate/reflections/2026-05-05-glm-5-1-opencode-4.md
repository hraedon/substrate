## Project State

Substrate is now substantially hardened. The critical bugs are fixed: the `sweep_expired_claims` crash was a guaranteed `KeyError` that no test exercised, and the two replay gaps meant any transition with `custom_fields` or any work item with `not_before` would produce drift. These were the highest-value catches.

The hook consumer is now production-viable: reconnect on connection loss, `threading.Event` for safe stop, dead-letter for unregistered handlers, per-hook savepoints so one bad handler doesn't poison the batch. The `hook_defaults.max_retries` wiring closes a gap where the JSON schema validated a field that was then silently discarded.

The `_prelocked_wi` optimization is a minor win (saves one `SELECT FOR UPDATE` round-trip per claim mutation) but its interaction with `_check_escalation` taught a sharp lesson: passing a stale pre-locked row to a second `append_event` after the first one bumps `next_event_seq` causes a `UniqueViolation`. Only the first append after a lock can safely use the pre-locked row.

What remains as debt: BC-022 (re-registration semantics), the `read_events` API's priority-based filter dispatch (only the first matching filter wins, others silently ignored), and the `query_work_items` f-string SQL builder. None are blocking.

## Friction and Improvements

The `_prelocked_wi` stale-seq bug was self-inflicted. I should have traced the full data flow more carefully before applying the optimization — the `next_event_seq` mutation inside `append_event` invalidates the cached row for any subsequent caller. A future improvement: instead of passing the raw dict, pass a wrapper that detects staleness (compare `next_event_seq` against DB value) or only cache the lock-guaranteeing fields, not the mutable ones.

The `poll_and_process_hooks` savepoint change required careful thought about psycopg3's transaction nesting semantics. The codebase would benefit from a brief comment in `_connection.py` documenting how `conn.transaction()` creates savepoints when nested, since this is a psycopg3-specific behavior that implementers may not know.

## Observations

The most striking thing about this session was finding a guaranteed crash (`sweep_expired_claims`) that had zero test coverage. The code path requires creating a claim, waiting for expiry, then calling sweep — a temporal pattern that doesn't naturally arise in unit tests. This suggests the project would benefit from a time-aware test helper (e.g., `time_machine` or explicit `UPDATE claims SET expires_at = now() - interval '1s'`) to exercise expiry paths without `time.sleep`.

The audit also surfaced a pattern: many of the medium-severity issues were "the JSON schema validates it but the code ignores it" — `hook_defaults` was the clearest example, but the unused `WORKFLOW_VERSION_ALREADY_REGISTERED` error code is the same pattern. Validation without enforcement is worse than no validation, because it gives a false sense of correctness.
