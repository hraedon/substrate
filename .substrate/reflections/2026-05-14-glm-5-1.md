# Session Reflection — 2026-05-14

**Work summary:** Resolved the last two open breadcrumbs (BC-140, BC-141). BC-141 fixed `_dict_contains` to match Postgres `@>` deep containment for nested JSON custom fields. BC-140 rewrote property tests to use `st.data()` iterative draw so idempotent event_id reuse actually fires. The BC-140 rewrite exposed three pre-existing InMemory atomicity bugs that were also fixed.

## On the project

The project is in excellent shape. Zero open breadcrumbs after this session. The conformance testing strategy (property-based, comparing Postgres and InMemory backends) is genuinely load-bearing — it caught real bugs that unit tests missed. The InMemory backend has been a consistent source of subtle divergence from Postgres transaction semantics, and I suspect there are more such issues hiding in less-exercised code paths (links, replay, hooks).

The `_contract.py` shared validation layer is elegant and works well. The `_event_store.py` extraction is clean. The main remaining risk is that InMemory is drifting toward being a second implementation rather than a thin test double — it's now ~1300 lines of parallel logic.

## On the work done

The BC-141 fix was straightforward — recursive containment is the obvious correct implementation, and the test workflow already had a `metadata` json field ready for testing.

BC-140 was more involved. The original `st.lists(operation())` approach was fundamentally broken: `seen_event_ids` was a dead parameter since hypothesis generates list elements independently. Replacing with `st.data()` was the right call. But the rewrite exposed a deeper issue: generating event IDs inside `_exec_op` meant each backend got a *different* random UUID, so the event pools diverged even with "fresh" IDs. Fixing this (generate at draw time, store in op dict) was essential.

The InMemory atomicity bugs were the most interesting finding. Postgres gives you transaction rollback for free; InMemory has no such mechanism. The pattern was consistent across three methods: mutate projection → append event → (if event fails, projection is corrupted). The fix was to defer all mutations until after the event succeeds. This is a design smell — ideally InMemory would have a transaction-like mechanism, but the try/except approach works for the cases we've found.

I'm confident in the `_dict_contains` fix. I'm less confident we've found all projection-before-event bugs in InMemory — the property tests exercise a limited operation surface (no links, no hooks dispatch, no replay-with-mutation).

## Gaps to flag

- **InMemory atomicity is whack-a-mole.** Three methods fixed this session, but the pattern (mutate-then-append) likely exists in other InMemory methods that the property tests don't exercise. A systematic audit of every method that calls `_store_append` or `_append_claim_event` or `_append_simple_event` would be worthwhile.
- **`_exec_op` generates `uuid.uuid4()` for operations without event IDs** (heartbeat, sweep). These are fine since they don't append events, but it's misleading — the fallback `or uuid.uuid4()` in `_exec_op` is dead code for ops that have `event_id` in the dict.
- **Stale `operation()` composite strategy** at the top of `test_property_conformance.py` is unused now — only `_draw_operation` is called. It adds ~70 lines of dead code.
- **Pre-existing uncommitted changes** from sessions 25 and earlier are in the working tree (BC-128 event store extraction, BC-139 custom field filters, AGENTS.md/spec.md updates). These should be committed as part of this session's commit.
