# Position on Debate 001 — Backend contract single-source-of-truth

**Property-based testing now. Defer contract extraction to the next significant feature that touches both backends.**

The diagnosis is correct: hand-maintained dual backends drift, and drift is discovered in production (golden runs) not in CI. Ten parity bugs (BC-045 through BC-058) is strong evidence. But zero parity bugs are currently open. The invariant is being maintained — expensively, but successfully.

**Why property-based testing is the right investment now:**

A `hypothesis`-based conformance test suite that generates random valid API call sequences and asserts state equivalence between InMemory and Postgres backends would:

1. Reduce detection latency from "golden run failure" to "CI failure within hours."
2. Cost ~1 session to implement (the API surface is bounded and well-typed).
3. Not require any changes to either backend's implementation.
4. Produce a living regression suite that catches the *next* parity bug automatically.

The debate mentions this as a "stopgap." It's not a stopgap — it's the correct long-term strategy for a system where the contract is "both backends produce identical state for all valid call sequences." Property-based testing is exactly the tool for that invariant.

**Why defer Option B (declarative contract):**

Option B extracts all state-transition logic into a single contract file that both backends interpret. This is architecturally correct but operationally heavy:

- InMemorySubstrate is 1,493 lines. The contract needs to capture all of it.
- The principal must be able to review the contract. A 1,493-line-equivalent contract file is not reviewable.
- The principal is a systems architect, not a developer. A declarative contract file is code, even if it's YAML or dataclasses.
- Building the contract is a large effort with no immediate user — it prevents future bugs but doesn't fix any current bug.

The right trigger: the next feature that requires modifying both backends significantly (e.g., multi-project links, advanced hook semantics, workflow composition). At that point, the contract pays for itself by reducing the modification surface from 2 files to 1. Building it preemptively is infrastructure without a consumer.

**Concrete next step:**

1. Add `hypothesis` to dev dependencies.
2. Create `tests/test_conformance.py` with strategies for each API call.
3. Generate random sequences: `create → claim → transition → release → sweep → replay`.
4. Compare final WorkItem, Event stream, Claim state, ReplayReport.
5. Run in CI on every PR + nightly.
