# Position on Plan 007 — Public API Facade Decomposition

## Position: Immediate cutover. No deprecation window.

We are the sole consumer of substrate. A deprecation window with 25+ `DeprecationWarning` wrappers adds noise and maintenance burden for zero benefit. The internal modules (`_workflow_api.py`, `_work_items_api.py`, etc.) are already extracted; this plan is purely about the public surface shape.

### Recommended execution

1. **Do NOT add deprecation warnings.** Rename old methods to aliases, log a single `info` event at import time if any legacy method is called (via a thin wrapper that increments a metric), but do not emit `DeprecationWarning`.
2. **Update all internal examples and tests to the new facade pattern in the same PR.** No mixed-mode codebase.
3. **Update the sidecar routes to use facades immediately.** The sidecar already does 1:1 mapping; it should consume the new API natively.
4. **Keep `transition()` and `replay()` on the top-level `Substrate` class.** These are orchestration-level operations, not domain-scoped. Moving them would create circular facade dependencies.
5. **Keep lifecycle methods (`create_project`, `close`, `connection_info`, `ensure_event_partitions`) on the top-level class.**

### Timing

Defer until **after Plan 008 (trust model hardening)** is implemented. Plan 008 benefits from per-domain policy attachment points (e.g., strict role checks on `WorkItemOps.create`), but 008 does not require 007 to function. Implementing 008 first means the facade objects are born with the right policy hooks already in place.

### Risk

| Risk | Mitigation |
|---|---|
| Agent code in v2 factory breaks on upgrade | We control v2. Update v2's substrate calls in the same commit that bumps the substrate version. |
| README / AGENTS.md drift | Update docs in the same PR. No mixed documentation. |
| `__init__.py` still large after facade extraction | Acceptable. It becomes a registry of facade properties + orchestration methods (~200 lines target). |
