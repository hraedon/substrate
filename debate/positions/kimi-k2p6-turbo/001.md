# Position on Substrate Debate 001 — Backend contract single-source-of-truth

**Agree with GLM and DeepSeek: property-based testing now, declarative contract only if the data justifies it.**

## Why the diagnosis is correct but the prescription may be wrong

Hand-maintained dual backends (Postgres + InMemorySubstrate) drift. This is fact, not speculation. Ten resolved parity bugs (BC-045 through BC-058) and two golden-run escapes (BC-063) prove it. The question is whether a declarative contract (Option B) is the right cure.

**DeepSeek's evidence critique lands cleanly:**
- 12 bugs total, all resolved
- 260 tests passing, 0 lint errors
- Zero open breadcrumbs

The bug rate may already be contained by the existing conformance test suite. Building a 1,493-line-equivalent contract file is chemotherapy for a headache if the headache stopped.

## Why property-based testing is the right first step

A `hypothesis`-based conformance suite that generates random valid API call sequences and asserts state equivalence between backends:
1. Requires no backend changes
2. Costs ~1 session to implement
3. Produces a living regression suite
4. Generates data: how many divergences per 10,000 sequences?

If `hypothesis` finds 10+ new divergences not caught by existing tests, the contract is justified. If it finds 0–1, the existing tests are adequate and the contract is over-engineering.

## Why I'd defer the contract even longer than GLM

GLM says: "build the contract when the next significant feature touches both backends." I'd go further: **build the contract only when `hypothesis` finds a divergence class that cannot be fixed by improving the existing conformance tests.**

The contract adds a third source of truth. A third source of truth creates two new divergence vectors:
- Contract/Postgres drift
- Contract/InMemory drift

The principal (systems architect, not developer) would need to review the contract. A YAML or dataclass contract describing SQL locking semantics (`SELECT FOR UPDATE`, gap-free `event_seq` allocation) is not reviewable without database expertise. The debate says "the contract is the thing the principal reviews" — but it isn't reviewable in its current design.

## What I'd do instead

1. **Month 1:** Add `hypothesis` to dev dependencies. Create `tests/test_conformance.py`.
2. **Month 1:** Run nightly for 30 days. Count divergences found.
3. **Month 2:** If divergence rate > 2/month → build contract. If ≤ 2/month → add more conformance tests, skip contract.

## What I'd push back on

The debate's acceptance criterion #1: "A human adding a new transition rule edits exactly one file." This is elegant but ignores the Postgres-specific semantics. Gap-free `event_seq` allocation under `SELECT FOR UPDATE` has no equivalent in a Python dict contract. The mapping layer becomes a new source of complexity, not a simplification.

## Connection to v2 BC-063

v2's golden-run escapes are the strongest argument for the contract. But those escapes were caught by integration tests (real substrate + real gates), not conformance tests. The fix was to improve the integration test surface (BC-063), not to build a contract. The contract would not have prevented BC-063's bugs.

## Bottom line

Don't build infrastructure to prevent a problem that may already be solved. Measure first. The measurement itself (property-based testing) is valuable regardless of the contract decision.
