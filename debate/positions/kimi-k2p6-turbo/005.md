# Position on Plan 009 — Operational Runtime (Substrate Daemon)

## Position: Implement Option A (built-in timer thread) only. Defer Option B (standalone daemon). Add maintenance metrics and a health indicator. Consider backoff-when-idle.

### Option A is sufficient

The sidecar (Plan 005) is already a daemon process. Consumers running the sidecar get maintenance there. Consumers embedding substrate in-process get `start_maintenance()`. A third deployment artifact (`substrate-maintainer`) is speculative infrastructure without a concrete consumer.

**Defer Option B until there is a deployment scenario where neither in-process embedding nor the sidecar is appropriate.**

### Subsume hook consumer

`start_maintenance()` should subsume `start_hook_consumer()` / `stop_hook_consumer()`. Having two separate threading APIs was always awkward. One call to start everything is the right UX.

### Add maintenance metrics

Plan 009 mentions `substrate_maintenance_errors_total` but does not specify operational counters for:
- `substrate_maintenance_sweeps_total` — claim sweeps executed
- `substrate_maintenance_claims_swept_total` — claims actually expired and swept
- `substrate_maintenance_partitions_created_total` — event partitions created
- `substrate_maintenance_recurrences_fired_total` — recurring work-items created
- `substrate_maintenance_hook_leases_swept_total` — stranded hook leases recovered

These are essential for operators to verify the maintenance thread is doing work, not just running no-ops.

### Add a health indicator

The plan mentions an `on_error` callback but does not provide a polling-friendly health signal. Add:

```python
@property
def maintenance_healthy(self) -> bool:
    """True if the maintenance thread is running and its last cycle succeeded."""
```

This lets consumers expose a health check without threading callbacks into their event loop.

### Consider backoff-when-idle

The default intervals (`sweep=30s`, `recurrence=10s`, `hook=2s`) create noise even when there's nothing to do. Consider exponential backoff when a cycle finds no work:

```
if claims_swept == 0:
    next_sweep_interval = min(current * 2, max_interval)
else:
    next_sweep_interval = base_interval
```

This reduces DB load for low-traffic projects without sacrificing responsiveness when work appears.

### Spec amendment precision

The proposed §3 amendment uses the word "runtime," which sounds like a daemon. Phrase it as:

> "Substrate provides optional background maintenance threads for timer-driven infrastructure operations. It does not require a separate daemon process for correctness."

This preserves the "library, not daemon" framing while acknowledging the thread.

### Sequencing

Implement **after** Plan 007 (facade decomposition), so maintenance methods (`start_maintenance`, `stop_maintenance`) live on a clean top-level class without crowding domain methods. If 007 is deferred, 009 can still proceed — the maintenance methods are already infrastructure-level.

### Connection to Plan 007

After facade decomposition, `start_maintenance` / `stop_maintenance` become clearer as the only top-level methods related to background infrastructure. The domain facades contain no timer methods. This is a clean separation.
