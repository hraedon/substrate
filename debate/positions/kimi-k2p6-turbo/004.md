# Position on Plan 008 — Trust Model Hardening for Multi-Tenant Deployment

## Position: Strong RFC. Implement WS-1 and WS-5 in the next session. Scope down WS-2. WS-3 needs CI-gated cross-validation.

### WS-1: Mandatory role enforcement (Priority 1)

The `strict_roles: bool = False` flag is the right backward-compatible migration path. The `role_source` check (reject `prompt`-source roles when strict) is a subtle but important protection against prompt-injection attacks claiming elevated roles.

**Accept as-is. Implement first.**

### WS-5: Key rotation safety (Priority 2)

Raising on unknown key status is a 3-line fix that closes an operational foot-gun. The `expected_key_count` assertion and `keys_loaded` structured log are cheap wins. 

**Accept as-is. Implement second.**

### WS-2: Key material protection (Priority 4, scoped down)

The `mlock()` and zeroization proposals are partially wishful thinking in CPython:
- `mlock()` is fragile in containers and best-effort only.
- Python string interning and the GC make zeroization unreliable.
- These provide security theater more than defense.

**Recommended scope:** Drop `mlock()` and zeroization. Keep only **environment-variable injection** (`SUBSTRATE_HMAC_KEY_<KEY_ID>`) with fallback to file. If memory-protected keys are genuinely required, the answer is a separate signing process (HSM / Unix socket delegation) — which correctly remains out of scope for this RFC.

### WS-3: rfc8785 supply-chain hardening (Priority 3)

Vendoring `rfc8785` is the right call. The cross-validation test must be **CI-gated** (a pytest test that runs on every PR), not just "at build time." Otherwise upstream drift will not be caught when the vendored copy is bumped six months later.

**Accept with amendment:** Make cross-validation a required CI test.

### WS-4: Sidecar rate limiting (Priority 5)

Correctly scoped to the sidecar. Consider using `limits` directly rather than `slowapi` to reduce dependency surface, but this is a minor preference.

**Accept as-is. Defer until sidecar has users who need it.**

### Missing: Key provenance logging

Add a structured log field `key_source: "file" | "env"` to the `keys_loaded` event. In a multi-tenant deployment, knowing whether a key was injected via environment (expected) or loaded from a shared file (riskier) is useful for audit.

### Sequencing

WS-1 → WS-5 → WS-3 → scoped WS-2 → WS-4.

### Connection to Plan 007

Plan 008 WS-1 benefits from facade decomposition: `WorkItemOps.create()` can carry a per-domain `strict_roles` override. But 008 does not require 007; implement 008 first.
